# ADR: OpenCode-First Goal Loop Runtime

Status: Accepted (Codex compatibility subsequently removed; OpenCode and Claude Code are the active command hosts)

Date: 2026-06-08

## Decision

OpenCode is the default runtime for built-in package profiles and runner-command execution. Codex remains a compatibility runner for hosts, projects, and workflows that still need the Codex command surface.

The package-owned OpenCode stack is cohesive and batteries-included:

- package-owned defaults: generated `.opencode` project configuration, profile defaults, command surfaces, agent/skill/prompt projections, permission policy, LSP configuration, and the singleton `pipeline-gateway` MCP server;
- candidate integration APIs: official OpenCode integration APIs are vetted for future native session work but are not runtime dependencies until package code actually needs them;
- included ecosystem code: DCP code, handoff/session capture, telemetry hooks, snippet/prompt helpers, background-agent patterns, memory/context helpers, and reusable OpenCode skill/prompt inventories selected by the package.

Pipeline-owned schedule, gate, goal, verification, and continuation state remains the source of truth. OpenCode plugin session state may project useful runtime context, but it is never the authoritative workflow state and must not decide whether a pipeline goal is complete.

Team mode is represented as explicit pipeline schedule graphs generated by the scheduler. OpenCode subagents and plugin hooks execute the graph, but the graph, dependencies, retries, gates, and acceptance evidence remain pipeline-owned.

## Rationale

OpenCode has native extension points that match the runtime this package is moving toward:

- agents and subagents provide the execution surface for delegated work;
- skills provide package and project instructions that OpenCode can discover from `.opencode/skills/`, `.claude/skills/`, and `.agents/skills/`;
- plugins expose event hooks for session, tool, and message lifecycle integration;
- MCP servers provide the shared tool surface through project configuration;
- LSP configuration gives the runtime diagnostics and navigation support from the host editor/tooling environment.

The package still needs its own scheduler and state model because the pipeline contract is stricter than a single OpenCode session. Goal persistence, continuation prompts, verification passes, acceptance evidence, and team-mode graph execution must survive host restarts, agent handoff, and mixed runner use.

## Library-First Vetting

| Item | Role in the Stack | Decision |
| --- | --- | --- |
| `@opencode-ai/plugin` | Official plugin API for OpenCode lifecycle hooks | Use when package code implements OpenCode plugins. |
| `@opencode-ai/sdk` | Official programmatic API for OpenCode integration | Use when package code needs OpenCode API access outside generated config. |
| `opencode-handoff` | Handoff/session capture pattern | Incorporate the relevant handoff behaviour into package-owned continuation flows. |
| DCP code | Dynamic context pruning for long OpenCode sessions | Include in the default OpenCode stack and wire it into continuation/context policy. |
| `@devtheops/opencode-plugin-otel` | OpenTelemetry plugin reference | Use as telemetry pattern input for package-owned runtime observability. |
| `opencode-snip` | Snippet/prompt helper pattern | Use as prompt-discipline and reusable snippet input. |
| `opencode-background-agents` | Background/delegated agent pattern | Use as delegation input for scheduler-generated team mode. |
| `opencode-mem` | Memory/context helper pattern | Use as input for durable goal context and memory projection. |
| `cupcake` | OpenCode ecosystem utility surfaced during research | Keep in the surfaced ecosystem inventory when relevant to package workflows. |

## Consequences

`PIPE-52.2` must switch built-in defaults and runner-command execution to OpenCode while preserving Codex compatibility paths.

`PIPE-52.3`, `PIPE-52.4`, and `PIPE-52.5` must define and enforce durable goal state, continuation prompts, and verification gates in pipeline-owned state rather than OpenCode plugin session state.

`PIPE-52.6` must generate team-mode schedule graphs that OpenCode can execute through agents/subagents while the pipeline remains responsible for graph state and acceptance evidence.

`PIPE-52.7`, `PIPE-52.8`, and `PIPE-52.9` must project the curated OpenCode stack into project resources and surface the selected ecosystem code, MCP servers, skills, prompts, permissions, LSP, and plugin hooks.

`PIPE-52.10` must keep runner capability handling behind an adapter so Codex compatibility does not leak into OpenCode-first defaults.

The first OpenCode adapter is `opencode-cli-subprocess`: it preserves the current `opencode run --format json` subprocess behavior, records session metadata from the launch plan, normalizes JSON event output, and explicitly marks continuation/session inspection APIs as unavailable. `@opencode-ai/sdk` and `@opencode-ai/plugin` remain documented candidates for later native session continuation, inspection, and plugin event integration rather than implicit dependencies of the CLI path.

`PIPE-52.12` must dogfood the OpenCode-first goal loop through real generated package surfaces and real pipeline usage.
